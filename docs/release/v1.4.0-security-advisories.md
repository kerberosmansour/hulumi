# v1.4.0 Security Advisory Registration Checklist

This checklist prepares the GHSA registrations for the v1.4.0 package release. **Publish
the advisories only after the `v1.4.0` release workflow has completed and the patched
npm tarballs are visible.** Drafts can be created at any time; publication is the
outward-facing step that must be gated on the patched tarballs actually being
installable.

GitHub's repository security advisory API is documented at
<https://docs.github.com/en/enterprise-cloud@latest/rest/security-advisories/repository-advisories>.
Check the current schema before scripting advisory creation; the table below is the
maintainer source of truth for titles, affected package ranges, patched versions, and
draft body text.

### GHSA-1: G_OIDC_1 + G_OIDC_2 array-`Federated` OIDC trust bypass

**Affected:** `@hulumi/policies` `< 1.4.0` — **Fixed in:** `1.4.0` — **Severity:** High — **CWE-697 (Incorrect Comparison)**

#### Summary

The G_OIDC_1 wildcard-OIDC guard and the G_OIDC_2 cluster-admin / `AdministratorAccess`
detector both used `federatedIsGithubOidc(String(principal.Federated))` to identify
trust policies that federate GitHub Actions OIDC. AWS IAM allows
`Principal.Federated` to be a JSON array; `String([...])` coerces the array to a
comma-joined string, the matcher extracts only from the first `oidc-provider/`
marker, and the exact-host comparison fails. The result: a trust policy that lists the
real GitHub OIDC provider ARN alongside any second federated provider bypasses both
detectors and can ship an `aws:iam:Role` that is genuinely GitHub-OIDC-assumable but
appears in stack validation as if it were not.

#### Impact

Downstream consumers of `@hulumi/policies`'s `HulumiHardeningPack` /
`HulumiGithubHardeningPack` could ship an IAM role with wildcard `sub` conditions
(G_OIDC_1) or with cluster-admin / `AdministratorAccess` (G_OIDC_2) while believing
the role's GitHub-OIDC trust path was being checked. A malicious downstream IaC change
could intentionally craft this shape to evade the mandatory wildcard-OIDC guard.

#### Patches

Upgrade to `@hulumi/policies@1.4.0`. The shared matcher in `github-oidc-issuer.ts` now
accepts `string | string[] | unknown` and matches per-array-entry; both call sites in
`g-oidc-1.ts` and `g-oidc-2.ts` pass `principal.Federated` directly (the lossy
`String(...)` coercion was removed).

#### Workarounds

None — upgrade is the only mitigation. Local-policy enforcement of "no array
`Principal.Federated` in IAM trust policies" is not a recommended substitute.

#### References

- PR #178 (Cluster A in the security PR).
- Anchored regression tests at `packages/policies/tests/github/{g-oidc-2,github-oidc-issuer}.test.ts`.

### GHSA-2: Forged-logical-name URN spoof in `urn.includes("<type>$")` policy gates

**Affected:** `@hulumi/policies` `< 1.4.0` — **Fixed in:** `1.4.0` — **Severity:** High — **CWE-693 (Protection Mechanism Failure)**

#### Summary

Multiple policy packs used `resource.urn.includes("<componentType>$")` to decide
whether a raw resource was a child of a trusted parent component. Pulumi URNs
include the operator-controlled logical resource name after the final `::`, so a raw
resource declared with a logical name embedding the parent-component type token
(for example: `github:index/repository:Repository::hulumi:platform:DeploymentRepositoryFoundation$deploy-repo`)
caused every such gate to skip its check. Affected packs (all in `@hulumi/policies`):

- `aws/hulumi-hardening-pack.ts` (H4 logging sibling lookup, H5 sibling lookup).
- `aws/cis-v5-pack.rules.ts` (CIS 2.1.1, CIS 2.1.5; advisory enforcement).
- `platform/deployment-governance-pack.ts` (`DEPLOY_GOV_1` — reported finding).
- `cloudflare/hulumi-hardening-pack.ts` (`CF_DNS_1`, `CF_DNSSEC_1` — unreported but exploitable; mandatory enforcement).
- `github/hulumi-hardening-pack.rules.ts` (GitHub-H1, GitHub-H2 — unreported but exploitable; mandatory enforcement).

#### Impact

Downstream consumers using any of the affected packs could ship a raw resource that
silently bypassed mandatory hardening checks via a forged logical name. Mandatory
(not advisory) rules: H1, H2, H4, H5 on AWS; H1, H2 on GitHub; CF_DNS_1, CF_DNSSEC_1
on Cloudflare; DEPLOY_GOV_1 on Platform.

#### Patches

Upgrade to `@hulumi/policies@1.4.0`. New `packages/policies/src/urn.ts` provides
`parseUrn` / `isUrnChildOfComponent` / `urnsShareParentComponent` helpers that anchor
matching against the URN's type-chain only (never the logical-name suffix). All six
prior call sites have been migrated.

#### Workarounds

None reliable — a fork that adds CI-level lint rejecting raw resources with logical
names containing `$` would catch the trivial form of the spoof but not crafted
variants. Upgrade is the recommended mitigation.

#### References

- PR #178 (Cluster B; the URN-anchoring refactor).
- Spoof-vector regression tests across the affected packs in `packages/policies/tests/`.

### GHSA-3: HULUMI-H5 decoy-sibling bypass

**Affected:** `@hulumi/policies` `< 1.4.0` — **Fixed in:** `1.4.0` — **Severity:** High — **CWE-284 (Improper Access Control)**

#### Summary

HULUMI-H5 is the defence-in-depth check that closes HULUMI-H1's `SecureBucket`-URN
exemption. The original implementation matched sibling hardening resources by type +
shared parent URN prefix only, never verifying that the sibling's `bucket` property
referenced the exempted bucket or that the bucket policy's `Resource` block covered
it. A malicious downstream IaC author could ship a forged
`hulumi:baseline:aws:SecureBucket` wrapper around a raw, unhardened
`<component>-bucket`, plus five decoy hardening siblings parented under the same
forged wrapper but targeting a _different_ bucket. H1's URN exemption skipped the raw
bucket; H5's sibling check found the decoys and reported no violation; the actually
exempted bucket stayed unhardened.

#### Impact

Downstream consumers using `HulumiHardeningPack` could ship a raw unhardened S3
bucket that appears to satisfy both H1 (exempted) and H5 (siblings present), but is
in fact missing every hardened-default control a real `SecureBucket` emits (PAB,
SSE-KMS, BucketOwnerEnforced, versioning, TLS-only policy).

#### Patches

Upgrade to `@hulumi/policies@1.4.0`. The H5 sibling matcher now requires both
anchored parent-component match (via `urnsShareParentComponent`) AND value-binding:
every sibling's `bucket` prop and the bucket policy's `Resource` ARNs must reference
the exempted bucket explicitly. Decoy regression tests in
`packages/policies/tests/hulumi-hardening-pack.test.ts`.

#### Workarounds

None — the exemption itself is the mechanism, so the only fix is the value-binding
check. Upgrade.

#### References

- PR #178 (Cluster B); supersedes PR #175 which addressed only the value-binding
  half on a stale base.

### GHSA-4: AccountFoundation audit-delivery bucket integrity weakened

**Affected:** `@hulumi/baseline` `< 1.4.0` — **Fixed in:** `1.4.0` — **Severity:** High — **CWE-1059 (Insufficient Technical Documentation / Behavioral Inconsistency)**

#### Summary

Three independent downgrade vectors against the AccountFoundation CloudTrail / AWS
Config delivery bucket integrity: (a) the startup-hardened tier hard-coded
`objectLock: false` on the delivery bucket, disabling Object Lock retention; (b)
`forceDestroy` was forwarded into the audit-delivery bucket unconditionally, so a
downstream stack with `logBucketForceDestroy: true` could purge CloudTrail / Config
log objects on `pulumi destroy`; (c) sandbox-tier `AccountFoundation` created its
internal log bucket with `tier: "sandbox"`, dropping the CloudTrail-Lake
`EventDataStore` (the only remaining immutable-audit mechanism after vector (a)).

#### Impact

Consumers using `AccountFoundation` could ship an AWS account whose CloudTrail /
Config audit evidence was deletable by any S3-delete-capable principal, while
believing the startup-hardened tier guaranteed tamper-resistance. Sandbox-tier
AccountFoundation deployments had no audit immutability at all (vectors (a) and (c)
compounding).

#### Patches

Upgrade to `@hulumi/baseline@1.4.0`. New function-keyed invariant in `SecureBucket`:
when the bucket actually backs audit delivery (i.e. `awsServiceLogDelivery.cloudTrail`
or `.config` is true), at the startup-hardened tier the constructor blocks
`forceDestroy: true`, emits the CloudTrail-Lake `EventDataStore` regardless of parent
tier, and adds a deny-`s3:DeleteObject` retention floor bucket-policy statement
scoped to the CloudTrail / Config history+snapshot prefixes (excluding the
`ConfigWritabilityCheckFile` probe key so AWS Config's write-then-delete probe is
unaffected). Object Lock is NOT re-enabled — the Config probe genuinely cannot work
with bucket-wide default retention.

#### Workarounds

Downstream consumers can replicate audit delivery to an Object-Locked archive
bucket outside the AccountFoundation tree as a partial mitigation while waiting to
upgrade. The recommended mitigation is upgrading.

#### References

- PR #178 (Cluster C); design rationale in CHANGELOG `### Migration`.

### GHSA-5: Drift classifier failed open on adapter failure and over-promoted Mixed verdicts

**Affected:** `@hulumi/drift` `< 1.4.0` — **Fixed in:** `1.4.0` — **Severity:** Medium — **CWE-755 (Improper Handling of Exceptional Conditions)**

#### Summary

Two symptoms of one root cause: the verdict snapshot was built from adapter
`detected` booleans only, discarding adapter `ok`/failure state and the CloudTrail
adapter's `ct.detected`. As a result: (a) an Automation-API adapter failure (e.g.
transient network) was unwrapped as `{ detected: false, ok: false }`, collapsed to
`mutated: false`, classified as `None / none`, and **written to the cache** for the
default 6-hour TTL — subsequent calls within the TTL skip every adapter, hiding any
real console-break-glass mutation; and (b) `Mixed` / `ConsoleBreakGlass` verdicts at
`high` confidence could be reached on healthy probe + provider drift alone, with
`ct.detected: false`, over-promoting normal provider churn to incident severity.

#### Impact

Consumers using `@hulumi/drift` for drift detection could see real console-break-glass
mutations silently swallowed (case a) or normal provider-version churn falsely
escalated (case b). Either way, the verdict source was unreliable for downstream
incident workflows that gate on it.

#### Patches

Upgrade to `@hulumi/drift@1.4.0`. Classifier-only fix (the TLA+-bound 6-row verdict
matrix in `tests/_utils/trace-matrix.ts` is byte-identical): a pre-`hardenedVerdict`
degradation gate fails closed to `Unknown / low` when any required adapter has
`ok === false` and the degraded verdict is not written to the cache. The Mixed /
ConsoleBreakGlass post-`hardenedVerdict` correction now uses `ct.detected` regardless
of probe state.

#### Workarounds

Set `options.minConfidence: "medium"` in the classifier call to prevent the
degraded `None / none` from being cached (it doesn't meet the threshold). Does not
fix case (b). Upgrade is the recommended mitigation.

#### References

- PR #178 (Cluster D); regression tests in
  `packages/drift/tests/classifier-fail-closed.test.ts`.

### GHSA-6: Detective-service reuse silently downgraded GuardDuty / Security Hub baseline

**Affected:** `@hulumi/baseline` `< 1.4.0` — **Fixed in:** `1.4.0` — **Severity:** Medium — **CWE-693 (Protection Mechanism Failure)**

#### Summary

`AccountFoundation`'s `existingGuardDutyDetectorId` reuse path used `Detector.get`
(read-only) and never asserted the imported detector was `ENABLED` with the baseline
`findingPublishingFrequency: FIFTEEN_MINUTES`. A suspended, weaker, or non-Hulumi
detector therefore satisfied the component's `guardDutyDetectorId` output without
the baseline actually being in effect. Separately, `useExistingSecurityHubAccount`
imported the account but created CIS / NIST `StandardsSubscription` resources with
default delete behaviour; a later `pulumi destroy` on the reused stack would
`BatchDisableStandards` CIS / NIST while leaving Security Hub itself enabled — a
silent monitoring downgrade of a pre-existing account.

#### Impact

Consumers reusing existing detective services with `AccountFoundation` could ship a
deployment that returns successful component outputs while the actual detective
controls are non-functional or are about to be silently disabled on next destroy.

#### Patches

Upgrade to `@hulumi/baseline@1.4.0`. GuardDuty reuse now asserts posture via
`getDetectorOutput(...).apply(...)` and throws on non-`ENABLED` /
non-`FIFTEEN_MINUTES`. Security Hub reuse creates the CIS / NIST subscriptions with
`retainOnDelete: true`. Net-new (non-reuse) paths unchanged.

#### Workarounds

Don't reuse existing detective services with `AccountFoundation` until upgraded.
Manually verify detector posture out-of-band if reuse is required.

#### References

- PR #178 (Cluster G); regression tests in
  `packages/baseline/tests/guardduty-reuse-posture.test.ts` and
  `packages/baseline/tests/securityhub-reuse-retain.test.ts`.

## Publication Steps

1. Confirm `.github/workflows/release.yml` succeeded for tag `v1.4.0` and the
   atomic six-package publish completed.
2. Confirm all six npm tarballs are visible:
   `@hulumi/baseline`, `@hulumi/policies`, `@hulumi/drift`, `@hulumi/k8s-baseline`,
   `@hulumi/cloudflare-baseline`, `@hulumi/platform-patterns`.
3. Run `pnpm run release:verify-attestations` against the published tarballs.
4. Create draft advisories in GitHub → **Security → Advisories → New draft security
   advisory**. For each row in the table above:
   - Ecosystem: `npm`.
   - Package name: as listed.
   - Affected versions: `< 1.4.0`.
   - Patched versions: `1.4.0`.
   - CWE: as listed (use the lookup).
   - Severity: as listed.
   - Description: paste the per-advisory body above (this file is the source of truth).
5. **Do NOT** request CVEs during initial draft. Decide on CVE request when
   publishing each advisory — the GHSA flow handles MITRE coordination if you opt in.
6. Publish advisories only after the npm artifacts are live and verifiable. The
   draft → publish step is the public-disclosure moment; consumers cannot patch
   until they can `pnpm update @hulumi/*` to 1.4.0.
7. After publication, GitHub will emit Dependabot alerts to consumers within minutes
   to hours. Confirm at least one alert appears (use a test consumer repo if
   convenient).

amendment to this checklist and treat the absence as a documentation gap, not a
