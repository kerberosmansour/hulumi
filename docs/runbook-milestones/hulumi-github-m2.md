# Milestone 2 — `@hulumi/baseline.github.OrgFoundation` + Code Security Configurations switchable backend

Parent runbook: [docs/RUNBOOK-hulumi-github.md](../RUNBOOK-hulumi-github.md). Read the runbook's Global Execution Rules (especially **Rule 0 — the infra-only scope contract**) + Global Entry Rules + [docs/lessons/hulumi-github-m1.md](../lessons/hulumi-github-m1.md) before starting.

**Goal**: After M2, `@hulumi/baseline.github.OrgFoundation` ships as a composing `ComponentResource` mirroring the AWS [`AccountFoundation`](../../packages/baseline/src/aws/account-foundation.ts) pattern. A single `new OrgFoundation(name, args)` provisions: an organization-level ruleset (signed-commits + deletion + force-push protection at startup-hardened tier), the Actions allowlist with the 2025-08-15 SHA-pin policy on at startup-hardened, the OIDC subject-claim customization template defaulted to the three-axis `repo → ref/environment → job_workflow_ref` safe shape (UNC6426 mitigation), and an encapsulated `securityDefaults` surface backed by **either** the legacy `OrganizationSettings` resource (`flat-fields` backend, default) **or** the GHAS Code Security Configurations REST surface (`code-security-configurations` backend, switchable). Switching backends is a one-line argument change with no public output-shape difference.

**Context**: Research [synthesis §"Forward-looking deprecation risk on flat org-settings fields"](../research/hulumi-github/synthesis.md) flagged that GitHub has marked the `PATCH /orgs/{org}` flat `*_enabled_for_new_repositories` fields with an "endpoint closing down notice. Please use code security configurations instead." `OrganizationSettings` writes those fields. If GitHub removes them before Hulumi-for-GitHub ships, every `OrgFoundation` deployment built directly on `OrganizationSettings` breaks. This milestone ships the abstraction *now* so the migration path is one config change, not a breaking API. The OIDC default is the second non-negotiable: UNC6426 (March 2026, ~500 vulnerable role ARNs across 275 AWS accounts per CSA Labs and Unit 42) weaponized exactly the wildcard `repo:{org}/{repo}:*` shape that Pulumi accepts as the provider's default. `OrgFoundation`'s default OIDC sub template is `repo:{org}/{repo}:job_workflow_ref:{org}/{repo}/.github/workflows/{workflow}@{ref}:environment:{environment}` with `:ref:refs/heads/main` documented as a fallback for repos without environments. Sub-component layout follows the AWS precedent (`account-foundation.ts` composes `cloudtrail.ts` / `config.ts` / `guardduty.ts` / `securityhub.ts` / `iam-baseline.ts` / `kms-ring.ts` without separate args/outputs files for each sub-component).

**Important design rule**: The `securityDefaults` surface is the **single point of encapsulation** for the GitHub flat-fields deprecation risk. `OrgFoundationArgs.securityDefaults` accepts a stable shape (`{ vulnerabilityReporting, secretScanning, secretScanningPushProtection, dependabotAlerts, dependabotSecurityUpdates, dependencyGraph, advancedSecurity }`) and `OrgFoundationArgs.organizationSecurityBackend` selects between `"flat-fields"` (default) and `"code-security-configurations"`. `OrgFoundationOutputs.securityDefaults` is identical regardless of backend choice — no consumer code branches on backend. Switching backends emits a structured `security_event.organization_security_backend_selected` log row to stderr with `{ org, backend, reason }` so audit logs can attribute the choice. **The CSC backend uses `pulumi.dynamic.Resource` for the REST escape hatch** — the existing [`packages/baseline/src/aws/probes/poll.ts`](../../packages/baseline/src/aws/probes/poll.ts) workaround for the vitest worker-pool gotcha is the documented pattern; the CSC backend's mock-runtime tests use `dependsOn` instead of dynamic resources for that reason.

**Refactor budget**: `Surgical addition only.` New `packages/baseline/src/github/org-foundation.ts` + sub-components. The only existing files modified are `packages/baseline/src/github/index.ts` (re-export `OrgFoundation` + types), `tests/skill-bdd/hulumi-threat-model.test.ts` if the four GitHub scenario exemplars (shipped in M1) need their forward-references updated to mention `OrgFoundation` is now available, and `docs/RUNBOOK-hulumi-github.md` Milestone Tracker.

## Out of Scope / Must Not Do

- No `HulumiGithubHardeningPack` — that's M3. M2 ships the safe defaults; M3 enforces them at preview-time via CrossGuard.
- No `G_OIDC_1` policy rule — that's M3. M2 ships runtime rejection of wildcard OIDC sub claims; M3 adds the declarative rejection.
- No `cis-github.ts` IDs mapping — that's M3 (gated behind WorkBench access).
- No drift classifier extension, no webhook adapter — that's M4.
- No SLSA-L3 release — that's M5.
- No `EnterpriseSecurityAnalysisSettings`, no audit-log REST polling, no audit-log streams — those are GHEC-only and v1.1 deferrals (per the runbook tier decision).
- No `hulumi:controls` tag on `OrgFoundation` outputs in M2 — same staged-migration as M1's `SecureRepository`. M3 adds it.
- No `acknowledgePublic`-equivalent for org-level operations — orgs don't have a "public visibility" axis; the equivalent risk surface is `defaultRepositoryPermission` which is constrained to safe values via the type union.
- No new `examples/`/`docs/cookbooks/` content — defer to M5 launch readiness.

5. Implement the four sub-components (`org-rulesets.ts`, `org-actions.ts`, `org-oidc-template.ts`, `org-security-defaults.ts`). Add the `assertOidcTemplateSafe` function next to `org-oidc-template.ts` (do NOT export from the package public surface — internal).
6. Implement `OrgFoundation` composing the four sub-components. The constructor `assertValidTier`s, instantiates the chosen backend, calls each sub-component with `{ parent: this }`, and emits the `security_event.organization_security_backend_selected` audit row.

8. Run `pnpm -r build && pnpm -r test && pnpm -r typecheck && pnpm -r lint && pnpm run lint:license-boundary && pnpm run lint:exact-pin-guard` — all green except the integration test (which skips without env vars).
9. Run integration test against the real sandbox org (manual, on a developer laptop with creds in env vars): once with `organizationSecurityBackend: "flat-fields"` and once with `"code-security-configurations"`. Confirm `OrgFoundationOutputs.securityDefaults.appliedFlags` is identical across the two runs and `configurationId` is set only on the CSC run. Verify teardown leaves the sandbox org clean.
10. Smoke tests + Self-Review Gate. Update Milestone Tracker to `done`, write lessons + completion files (record any vitest-pool gotchas with the dynamic-resource CSC backend honestly).

## Notes

- This milestone ships **no** policy pack, **no** drift adapter — those are M3 / M4. M3's `HulumiGithubHardeningPack` H1 will reject any `OrgFoundation` constructed with a wildcard custom OIDC template at preview-time (declarative rejection); M2 ships the runtime rejection so the discipline holds even without the pack loaded.
- The CSC backend's `pulumi.dynamic.Resource` is the most operationally fragile surface in this milestone. Lessons file MUST record any vitest-pool surprises, REST rate-limit hits during integration runs, and any deprecation-timeline updates GitHub publishes between M2 start and M2 end. The pulumi-version matrix test (added per critique E2) catches dynamic-resource-semantics drift in `@pulumi/pulumi` minor bumps.
- The `hulumi:controls` tag remains deliberately omitted on `OrgFoundation` outputs (same staged-migration as M1's `SecureRepository`). M3 adds it as an additive change.
- Backend-swap parity test is a load-bearing invariant: if the two backends ever produce divergent `appliedFlags`, the encapsulation has leaked and the `OrgFoundationOutputs.securityDefaults` shape contract is broken. Treat this as a release-blocking regression.
- The token-redaction layer (per critique S2) is a non-trivial implementation choice: regex-strip vs. structured-log-with-typed-fields. M2 lessons file MUST record which approach was chosen and why; M5 launch-readiness verifies the cookbook reference doc cites the chosen pattern.
- Backend swap continuous-protection (per critique S3) depends on Pulumi's `dependsOn` semantics. If `dependsOn` cannot prevent the destroy-before-create gap (Pulumi engine specifics), M2 documents a maintenance-window requirement in the cookbook (M5) rather than silently accepting the gap.
