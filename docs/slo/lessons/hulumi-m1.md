## Design decisions and why

- **`.mjs` scripts, not `.ts`.** The M1 contract forbids runtime dependencies and the skill is meant to be `git clone`-installed into `~/.claude/skills/` on any Node 20 machine. Shipping `.ts` files would require `tsx` or a build step at install time, adding a runtime dep surface users would then need to trust. Plain `.mjs` + JSDoc types gives editor-surface type safety without the runtime-tool dependency.
- **Scripts are `esmodule` with a `isMainModule` guard.** Same module serves as a library import for tests (`import { generateThreatModel } from "…/generate-threat-model.mjs"`) and a CLI invocation for users (`node generate-threat-model.mjs <scenario>`). Common Node pattern; avoids duplicating logic.
- **License-boundary lint scans `skills/` + `packages/` only, not `tests/`.** Tests contain deliberate fixture files with CCM-opener fragments (to test that the lint catches misuse). Scanning tests would create unresolvable false positives.
- **Framework naming uses the versioned form `CIS-AWS-v5.0.0` in IDs + `requiredFrameworks` + mapping file entries.** Initially I named the framework `CIS-AWS` and used the versioned form only in IDs — this created a mismatch in citation lookup. Normalizing to `CIS-AWS-v5.0.0` everywhere is less ambiguous when CIS v7.0.0 lands as a second framework in v1.1+.
- **Prettier ignores `skills/hulumi-threat-model/templates/`.** Prettier reflows markdown tables and inserts a blank line between the header-separator and the `{{STRIDE_ROWS}}` / `{{CITATION_ROWS}}` placeholders — which breaks the rendered markdown table because that blank line terminates the table parse. `<!-- prettier-ignore -->` didn't help; a `.prettierignore` entry did.
- **`renderComponents` is defensive against missing `recommendedComponents`.** The BDD suite's synthetic-empty scenario JSON omits that field; the generator emits a graceful "no Hulumi components recommended yet" line rather than crashing. Cheap, resilient, documents a genuine empty-state outcome.
- **Planning-artifact links in the migrated runbook annotated, not copied.** M1 allow-list only permits migrating the runbook + milestones. The idea/research/design/TLA+/critique docs live in the upstream planning corpus; the migrated runbook's Prerequisite reading section lists them by path + notes they aren't in this repo yet. M5 follow-up tracks whether to import them.

## Template improvements suggested for the runbook-template_v_3

- **"Files to read before changing anything" shouldn't silently expand the write allow-list.** Several files in M1's read-list are in the upstream TauriMobile repo, not this repo. The template's "allowed to change" column is explicit about repo-relative paths, but "files to read" is ambiguous about cross-repo locations. Add a column or convention for "repo-of-origin" in both read-list and allow-list tables.
- **For greenfield milestones, the "Baseline tests" Evidence Log row needs an explicit "N/A — greenfield" option.** The current template assumes a pre-existing suite; forcing "(none — record in Evidence Log)" works but isn't discoverable.

## Rules for the next milestone (M2)

- **`@pulumi/*` deps land in M2.** Exact-pin with `pnpm-lock.yaml` integrity hashes from day one. Commit the lockfile. Capture the pinned versions + integrity hashes in the M2 Evidence Log for later cross-reference from the M5 cooling-off CI check.
- **Prettier will break any new markdown tables with placeholders.** M2's `docs/tiers.md` tier-matrix table is static (no `{{…}}` placeholders) so it's safe, but if M2 adds any templated markdown with tables, add to `.prettierignore` immediately.
- **ESLint 9 flat config has narrower globals by default.** Any new `.mjs` or `.ts` file using Node globals (`process`, `console`, `__dirname`) needs the global to be listed in `eslint.config.mjs` or the file scopes its own. Applied lesson: globals block in place for both TS and JS files.
- **Defensive JSDoc for Record<string, …> lookups.** `BUNDLED_STUBS[someStringVar]` trips TS's strict indexing check when the type is a struct of literals. Annotate `BUNDLED_STUBS` as `Record<string, …>` up front when the key space will be dynamic.
- **`/** @returns {never} \*/`is required on functions that`process.exit()`.** Otherwise TS won't narrow `args.scenario`from`string | undefined`to`string` after the guard. Add the annotation to every exit-path helper.
- **The skill's output-schema lock means the M2 scenario JSON edits (drop "v0.2+" forward-references for `SecureBucket`) must NOT change the frontmatter or section structure.** Only strings inside existing sections.

## /slo-verify M1 observations (appended 2026-04-24)

- **No new bugs found** during runtime verification. Full verify report: [docs/slo/verify/hulumi-m1.md](../verify/hulumi-m1.md).
- **Two deliberate-but-surprising behaviours surfaced** and are documented in the verify report so future readers don't mistake them for bugs:
  - `printUsageAndExit()` emits the usage block to both stdout and stderr (so the BDD test can read scenarios from either stream). Running the CLI in a terminal displays the help message twice.
  - `pnpm install` emits a `DEP0169 url.parse()` deprecation warning on Node 24.x (`@pnpm/*` internals, not our code). Doesn't emit on the Node 20 LTS target.
- **Coverage gap carried forward**: the demo gate (installing the skill into `~/.claude/skills/hulumi-threat-model/` and invoking in a fresh Claude Code session) was not exercised in /slo-verify. Direct CLI invocation exercises the same entry points Claude Code would hit, so the gap is narrow — but the first real install-and-invoke session before M5 npm publish is still worth doing as a manual sanity check.

## Template improvements suggested by /slo-verify + /slo-retro

- **`docs/slo/verify/`, `docs/slo/lessons/`, and `docs/slo/completion/` are standard post-milestone artifacts but are NOT in M1's `Files Allowed to Change` table.** The v3 runbook template treats these as implicit, but the allow-list table doesn't call them out. Suggestion: add a standing "runbook-metadata artifacts" exemption to the Global Execution Rules so every milestone inherits write access to `docs/{verify,lessons,completion}/<prefix>-m<N>.md` without needing to list them per milestone.
- **Prettier keeps biting on markdown tables with glob patterns in prose** (`*.foo`, `.env`, `_underscored_names`). The Evidence Log row for the `.gitignore` check had to be rewritten to avoid raw `*.tsbuildinfo`, `.sigstore`, `.env`, `*_TTrace_*` because prettier's `--check` fails idempotency on them. For M2 and onward, wrap any file-glob pattern in single backticks inside Evidence Log cells, or describe patterns in prose without raw glob characters.
