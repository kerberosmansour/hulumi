## Initial Findings

- **Pulumi GitHub provider exists and is first-party (`pulumi/pulumi-github`, bridged from `integrations/github` Terraform provider).** Confidence: **high** that it exists and is bridged; **needs verification** for current 2026-04-25 version, and for which of {`RepositoryRuleset`, `OrganizationRuleset`, GHAS security configurations, code-security-configuration attachment, audit-log streaming destinations, push-protection/secret-scanning enablement} are first-class vs. require REST escape hatches.
- **Terraform `integrations/github` is the upstream the Pulumi provider bridges from**, so historical parity has been near-1:1 with a lag (Pulumi typically a release behind on new resources). Confidence: **medium**; the lag size in 2026 needs verification.
- **Repository rulesets are the modern replacement for legacy branch protection.** GitHub introduced rulesets (repo + org level) in 2023; legacy `branch_protection_v3`/`branch_protection` still works but rulesets are the GA path forward. Confidence: **high** for the existence; **needs verification** for 2026 provider coverage and any deprecation timeline on legacy branch protection.
- **GitHub Advanced Security "security configurations" / "code security configurations" object** is a 2024+ scoping primitive that lets orgs attach a named bundle of GHAS settings to repos. Confidence: **medium**; provider coverage in 2026 needs verification.
- **Auth modes for `pulumi-github`:** classic PAT, fine-grained PAT, and GitHub App installation token are all supported by the Terraform provider; Pulumi inherits this. GitHub's own 2024+ guidance steers IaC-execution-role auth toward GitHub Apps (installation tokens, scoped, rotatable, auditable per-installation) over PATs. Confidence: **medium-high** for the support; **needs verification** for which the docs explicitly steer toward in 2026.
- **Audit-log API tiering:** programmatic access to the org-level audit log REST API has historically required **GitHub Enterprise Cloud**; Free/Team orgs see only the UI audit log with limited retention. Audit-log streaming (Splunk/Datadog/Azure Event Hubs/AWS S3) is **GHEC-only**. Confidence: **medium-high**; needs 2026-04-25 verification because GitHub has been opening surfaces (e.g., security overview, dependency-review) to non-Enterprise tiers piecemeal.
- **`github/safe-settings`** is a maintained Probot from GitHub itself that applies declarative org-level repo settings via PR. License: ISC. Confidence: **high** that it exists; **needs verification** for current 2026 maintenance status, and for whether it supports rulesets / GHAS security configurations.
- **`ossf/allstar`** is OSSF's policy-enforcement bot for GitHub orgs (Apache-2.0). Different shape from Hulumi: it watches and reconciles, not declares. Confidence: **high** for the project; **needs verification** for current 2026 status (allstar maintenance has been intermittent).
- **`step-security/secure-repo`** is a hosted SaaS with an OSS GitHub App that hardens repo and Actions configuration. License: mixed (OSS components Apache-2.0; service is commercial). Confidence: **medium**; needs 2026 status verification.
- **`chainguard-dev/octo-sts`** is an OIDC trust broker (Apache-2.0) — different category from "hardened repo IaC" but adjacent because it replaces classic-PAT auth from CI to GitHub. Confidence: **medium-high**; current 2026 status needs verification.
- **`mineiros-io/terraform-github-repository`**: mineiros.io shut down in 2023, modules dormant. Confidence: **medium**; verification needed because community forks may have continued.
- **`philips-labs/terraform-github-repository`** and **`cloudposse/terraform-github-repository`**: both historically maintained. Cloudposse is Apache-2.0; Philips Labs is typically Apache-2.0. 2026 status needs verification.
- **CIS GitHub Benchmark** exists and is governed by the same CIS Benchmarks license posture as CIS AWS Foundations (free PDFs for non-commercial use; embedding control text in distributed product requires commercial license). ID-only citation is the same defensible posture. Confidence: **medium-high**; **needs verification** that the CIS license terms have not diverged for the GitHub Benchmark in 2026, and confirmation of current version number.
- **NIST SP 800-218 SSDF** practices map to several GitHub controls (PO.1.1 process docs, PS.1.1 source integrity, PS.2.1 code review, PW.7.1 code review, PW.8.1 vulnerability response). The mapping is published in informal practitioner write-ups (GitHub's own 2024 "Securing your enterprise"); a NIST-authored mapping document exists in draft. Confidence: **medium**; needs verification of whether NIST has published a final SSDF→GitHub mapping by 2026-04-25.
- **SLSA Source Track L1/L2/L3** specifies provenance and integrity properties of a source (not just build) repository; L3 requires two-person review and protected branches with logged history. SLSA Source Track moved from draft to v1.0 sometime in 2025 — verification needed for 2026 status. Confidence: **medium**.
- **OpenSSF Scorecard** mixes configuration-driven checks (`Branch-Protection`, `Token-Permissions`, `Pinned-Dependencies`, `Webhooks`) with analysis-driven checks (`SAST`, `Vulnerabilities`, `Fuzzing`). The configuration-driven set maps cleanly to declarable Pulumi components. Confidence: **high** for the split shape; **needs verification** for the current 2026 check list.
- **MITRE ATT&CK for SCM** — there is no formal ATT&CK matrix for SCM platforms as of training data; community matrices (e.g., Cycode "DevOps ATT&CK", Step Security threat catalogues) exist but are not MITRE-authored. Confidence: **medium**; verification needed for 2026.
- **GitHub Public Roadmap + Changelog blog tag feed** are the closest equivalents to `@pulumi/aws` release notes; there is no machine-readable diff of API rename events. The OpenAPI spec at `github/rest-api-description` provides the closest authoritative API-surface diff. Confidence: **medium-high**.
- **Threat-model scenario demand (priors, needs verification):** highest demand-minus-supply candidates appear to be (c) Actions supply-chain (post-tj-actions/changed-files March-2025 incident), (d) OIDC trust to cloud (heavily marketed but practitioner playbooks thin), (f) GitHub App / installation-token exposure (under-served), and (g) GHAS configuration (new feature, little public guidance). Org bootstrap and repo hardening are saturated by `safe-settings` walkthroughs. Self-hosted runners are well-covered by Datadog/StepSecurity blog posts. Confidence: **low-medium**, intentionally — this is the question to validate, not assume.
