### Q1 — Pulumi GitHub provider, gaps and auth (deepened)

- **Auth modes (H)**. `github.Provider` exposes exactly two mutually exclusive credential paths: `token` ("OAuth token used to connect to GitHub. Anonymous mode is enabled if both `token` and `appAuth` are not set") and `app_auth` ("The GitHub App credentials used to connect to GitHub. Conflicts with `token`"). There is **no separate "fine-grained PAT" property** — fine-grained PATs are passed through the same `token` field; the provider does not distinguish them at config-time ([github.Provider — Pulumi Registry](https://www.pulumi.com/registry/packages/github/api-docs/provider/), [Installation & Configuration](https://www.pulumi.com/registry/packages/github/installation-configuration/)).
- **Recommended least-privilege auth (M)**. Provider docs do **not** explicitly steer toward `app_auth`. The only auth example in the public Installation page is `GITHUB_TOKEN` with a `--secret` flag; the README's auth section similarly only documents PAT. Treat "GitHub App is the recommended IaC role" as a **community/Hulumi-side opinion, not a documented Pulumi-provider stance**.
- **Confirmed first-class resources (H)** — repository rulesets (`RepositoryRuleset`, `OrganizationRuleset`) including the `requiredCodeScanning` rule, `OrganizationSettings`, `ActionsOrganizationPermissions` (Actions allowlist), `ActionsOrganizationSecret`, `ActionsOrganizationSecretRepositories`, `ActionsRunnerGroup`, and **`ActionsOrganizationOidcSubjectClaimCustomizationTemplate`** (controls the `sub` claim shape that downstream cloud trust policies match against) ([Pulumi Registry — github package](https://www.pulumi.com/registry/packages/github/), [ActionsOrganizationOidcSubjectClaimCustomizationTemplate](https://www.pulumi.com/registry/packages/github/api-docs/actionsorganizationoidcsubjectclaimcustomizationtemplate/)).
- **Confirmed gaps that force escape hatches (H/L)**:
  - **GHAS "code security configuration" objects** (the new GHAS scoping objects that attach a security config to N repos): no resource of that name surfaced via the Pulumi Registry index, the API-docs index, or `code_security_configuration` / `repository_security_configuration` keyword searches against the provider source. **Treat as a Pulumi-provider gap as of v6.12.2 (2026-04-10), confidence M (negative-surface inference).** The Terraform provider also at v6.12.x does expose related security-and-analysis fields on `github_repository` but the *configuration-object-scoped-to-many-repos* GHAS resource is not surfaced there either in the search slice.
  - **Audit-log streaming destinations** (Splunk / Datadog / S3 / Azure Event Hubs / GCS / Azure Blob): no `OrganizationAuditLogStream*` or equivalent resource exists in the Pulumi GitHub provider — REST API surface is `/enterprises/{enterprise}/audit-log/streams` and Pulumi has no first-class binding to it ([Streaming the audit log — GHEC](https://docs.github.com/en/enterprise-cloud@latest/admin/monitoring-activity-in-your-enterprise/reviewing-audit-logs-for-your-enterprise/streaming-the-audit-log-for-your-enterprise)). Confidence M.
  - **GitHub-as-IdP declarations on the AWS/Azure/GCP side** are out-of-provider by design — the OIDC trust policy (the IAM role / federated credential / workload-identity-pool) lives in the cloud provider, not in pulumi-github. The github-side surface is the OIDC subject-claim customization resource named above.
- **Terraform parity (M)**. `integrations/terraform-provider-github` v6.12.0 (2026-04-23) matches Pulumi's coverage on the major resources; one search hit suggests Terraform exposes per-repo security-and-analysis fields including `advanced_security` toggles in the resource itself, while Pulumi's `github.Repository` similarly exposes a `security_and_analysis` block. Strict resource-by-resource diff against Terraform was not exhaustively verified in this slice.

### Q3 — Infra-vs-appsec boundary (deepened)

**Sourced framework anchors**:
- **NIST SSDF (SP 800-218 R1.2 draft, Dec 2025)**. GitHub's own Well-Architected mapping (Jan 2026) classifies SSDF practices as either **Infrastructure-as-Code** (PO.2 teams/CODEOWNERS, PO.3 security-config/Dependabot/secret-scanning, PO.4 rulesets, PO.5 environments+OIDC, PS.1 SSO/IP-allowlist/rulesets, PW.4 Actions policies/Dependabot/OIDC, PW.5 code-scanning rulesets, PW.6 runners, PW.7 PR rulesets+CODEOWNERS, RV.1 Security Overview/audit-log-streaming) or **Workflow** (PS.2 attestations, PW.1/PW.2 design docs, PW.8 DAST/container scans, RV.2 alert remediation, RV.3 Discussions). This is the cleanest published infra-vs-workflow split available ([NIST SSDF on GitHub — Well-Architected](https://wellarchitected.github.com/library/scenarios/nist-ssdf-implementation/), [NIST SP 800-218](https://csrc.nist.gov/pubs/sp/800/218/final)).
- **SLSA Source Track v1.2 (H)**. L1 (version-controlled URI), L2 (immutable history, branch updates restricted to descendants — i.e. force-push prevention, source-provenance attestations), L3 (continuous technical controls, tag immutability), L4 (two-party review). **L1–L3 are entirely repo/org configuration. L4 mixes configuration (require-N-reviewers) with per-PR review behavior** ([SLSA v1.2 source-requirements](https://slsa.dev/spec/v1.2/source-requirements)).
- **OpenSSF Scorecard (H)**. `Branch-Protection`, `Pinned-Dependencies`, `Token-Permissions`, `Webhooks` are configuration-driven; `Code-Review`, `Maintained`, `SAST`, `Vulnerabilities` are analysis-driven ([scorecard.dev](https://scorecard.dev/)).
- **CIS GitHub Benchmark v1.2.0** (March 2026 cohort, covers GitHub up to v3.18). Specific section text was not retrieved directly; community references confirm it is structured around branch protection, secret-scanning enablement, audit-log review, member/team management — i.e. organization configuration ([CIS Benchmarks March 2026 update](https://www.cisecurity.org/insights/blog/cis-benchmarks-march-2026-update)).
- **MITRE ATT&CK (M, partial)**. T1195 (Supply Chain Compromise) and sub-techniques T1195.001 / T1195.002 are the canonical "SCM-adjacent" techniques in 2026; ATT&CK does not have a dedicated "GitHub" or "SCM platform" technique in 2026 ([T1195](https://attack.mitre.org/techniques/T1195/)). Confidence: there is no SCM-specific ATT&CK matrix as of the search slice.

**Boundary positions (sourced)**:
- (a) **Enabling CodeQL/Dependabot/secret-scanning/push-protection at repo or org level → infra scope.** SSDF Well-Architected maps this to PO.3 / PW.5 and labels it Infrastructure-as-Code; GitHub Docs treats "Managing security and analysis settings" as the configuration plane ([Managing security and analysis settings](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/enabling-features-for-your-repository/managing-security-and-analysis-settings-for-your-repository)).
- (b) **Authoring CodeQL queries / Semgrep rules / custom secret-scanning patterns → appsec scope (out-of-scope).** SSDF Well-Architected maps PW.5 *enablement* to IaC but rule authoring lands in PW.5/PW.7 review behavior, not configuration; GitHub Docs splits "Quickstart for securing your repository" (workflow) from settings management.
- (c) **SHA-pinning Actions → split**. **Org-level SHA-pin enforcement is infra** (the GitHub Actions org policy GA'd 2025-08-15 supports SHA-pin enforcement as a setting — [Changelog](https://github.blog/changelog/2025-08-15-github-actions-policy-now-supports-blocking-and-sha-pinning-actions/)). **Per-workflow source review of pinned SHAs is appsec/workflow.** The 2026 `dependencies:` workflow lockfile is roadmap, not GA — design must not depend on it.
- (d) **Required workflows / org-required status checks → infra scope.** Configurable via rulesets and ActionsOrganizationPermissions; SSDF PW.7 IaC.
- (e) **GHAS security-configuration objects → infra scope** (but currently a Pulumi gap — escape hatch needed). SSDF PO.3 IaC.
- (f) **Triaging Dependabot/CodeQL alerts → appsec scope (out-of-scope).** SSDF RV.2 Workflow.

## Library Evaluations

### `pulumi-github` v6.12.2 (2026-04-10) — Apache-2.0
**Pros**: First-class `RepositoryRuleset`, `OrganizationRuleset`, `OrganizationSettings`, `ActionsOrganizationPermissions` (allowlist + SHA-pin block enforcement matching the 2025-08-15 platform GA), `ActionsOrganizationOidcSubjectClaimCustomizationTemplate` (lets a Pulumi program shape the `sub` claim that downstream IAM trust policies match — directly addresses the UNC6426 misconfiguration class). Built-in `app_auth` config block. Tracks Terraform parity at 6.12.x.
**Cons**: No GHAS code-security-configuration object resource; no audit-log streaming destination resource; provider docs do not steer toward GitHub-App auth as default — Hulumi must own that opinion. Fine-grained PAT distinction is implicit (same `token` field), so SecureRepository's prevent-classic-PAT story must be enforced at the Hulumi component layer, not the provider's.
**Fit**: Mandatory baseline. The two gaps (GHAS configs + audit-log streams) are the only places `local.command` / direct REST escape hatches will be needed, and they are the same gaps Terraform has — so this is structural to the platform, not Pulumi-specific.

### `octo-sts/app` (Chainguard) — Apache-2.0 (per Chainguard convention)
**Pros**: Replaces long-lived PATs and even installation tokens with OIDC-exchanged short-lived tokens. Directly attacks the (f) GitHub-App-token-exposure threat scenario.
**Cons**: A separate GitHub App that must be installed; not a Pulumi resource. Operational dependency, not a library.
**Fit**: Reference architecture / cookbook entry rather than vendored code. SecureRepository can document and emit guidance toward octo-sts as a hardened auth pattern without depending on it.

### `ossf/allstar` v4.5 (2025-10-01) — Apache-2.0
**Pros**: Active, mature detective control with a known org-level deployment story.
**Cons**: Detective (issues filed on violation), not preventive (rejects bad config at preview-time). Hulumi's wedge is preventive + declarative + drift-classified.
**Fit**: Complementary, not competitive. Document as a "second line of defense" Hulumi customers can layer; do not vendor.

### `github/safe-settings` 2.1.20-rc.3 (2026-03-31) — ISC
**Pros**: Active, GitHub-published, ships rulesets coverage including 2.1.x's branch-protection-as-code-via-YAML. Known to the same audience.

**Fit**: Direct competitor on the bare "settings as code" axis but stops there. Hulumi-for-GitHub differentiates on the policy pack + drift classifier + threat-model skill triple.

### `step-security/secure-repo` v1.12.0 (2026-04-17) — **AGPL-3.0**

## Architecture Options

### Option A — Three-package mirror of the AWS pattern
Add `@hulumi/baseline.github.SecureRepository`, `@hulumi/baseline.github.OrgFoundation`, `@hulumi/policies/github/packs/hulumi-hardening`, and `@hulumi/drift/adapters/github-audit-log` strictly mirroring the AWS structure. Tier enum reused (`sandbox` / `startup-hardened`). Hardening pack lives in a new file alongside the AWS hardening pack (one-PolicyPack-per-process constraint preserved).
**Trade-offs**: highest internal consistency, lowest cognitive load for existing users; OrgFoundation must internally compose RepositoryRuleset + OrganizationRuleset + OrganizationSettings + ActionsOrganizationOidcSubjectClaimCustomizationTemplate + ActionsOrganizationPermissions, plus an *escape-hatch sub-resource* for GHAS code-security configurations and audit-log streams (`local.command` or a small dynamic-provider wrapper around the REST API). Drift classifier degrades on Free/Pro/Team — design must surface a `tier-degraded` verdict explicitly.
**Best fit**: a customer who already runs Hulumi-AWS and wants the same idioms; a GHEC org that can use the audit-log adapter end-to-end.

### Option B — Two-package "infra-only contract" with explicit tier split
SecureRepository + OrgFoundation in `@hulumi/baseline.github`, but split the drift adapters into `@hulumi/drift/adapters/github-audit-log-ghec` (full fidelity) and `@hulumi/drift/adapters/github-webhook-fallback` (Team-tier, lower fidelity, subscribes to repo `branch_protection_rule` / `secret_scanning_alert_location` etc. webhooks). Component type strings: `hulumi:baseline:github:SecureRepository`, `hulumi:baseline:github:OrgFoundation`.
**Trade-offs**: one more adapter to maintain, but the design "owns honestly" the tier-degraded experience for non-Enterprise customers, which most Hulumi-AWS users will be (a small-medium company on GitHub Team, GHEC on AWS — that's the wedge persona). The webhook adapter sidesteps the audit-log-API auth-mode constraint (no PAT/installation-token issue) because webhooks are push.
**Best fit**: the canonical persona in the brief — platform engineer with Hulumi-AWS adopted, ~30 repos, hand-configured drift, **probably not GHEC**. If validated with the user that they are non-Enterprise, this is the better default.

### Option C — Thin baseline + heavy policy pack
SecureRepository is minimal (rulesets + security-and-analysis + Actions allowlist); the **HulumiGithubHardeningPack does the heavy lifting**, blocking raw `github.Repository` without `securityAndAnalysis`, blocking classic-PAT auth at the provider level (Policy AST inspection of `provider.token` source), blocking required-status-check absence, requiring `ActionsOrganizationOidcSubjectClaimCustomizationTemplate` to declare a non-default `repository_owner:org/repo` shape (UNC6426 mitigation enforced at preview-time).
**Trade-offs**: matches the AWS pattern's "H1 blocks raw S3 bucket / H2 blocks unencrypted state backend" idiom (per `packages/policies/src/index.ts`). Policy pack has the strongest enforcement story but creates more pull on Hulumi-tagged exemptions for legitimate edge cases. Threat-model skill scenarios bind directly to policy IDs.
**Best fit**: customers who want the tightest "no-escape-hatch" preview-time gate — particularly the threat-model-driven persona who needs the `/hulumi-threat-model` skill scenarios (d, c, f, e) to map cleanly to policy rule IDs.
