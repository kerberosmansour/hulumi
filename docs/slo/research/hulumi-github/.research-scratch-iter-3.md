### Q3 — Audit-log auth-mode constraint (Δ FULLY RESOLVED, H)

This was the highest-impact open question for the drift-classifier architecture. Direct quote from the GitHub Enterprise Cloud REST docs ([REST API endpoints for enterprise audit logs](https://docs.github.com/en/enterprise-cloud@latest/rest/enterprise-admin/audit-log)):

> "These endpoints only support authentication using a personal access token (classic)."

This applies uniformly to **all seven endpoints** in the family — including read (`GET /enterprises/{enterprise}/audit-log`), stream listing (`GET .../streams`), stream creation/update/delete (`POST/PUT/DELETE .../streams/{stream_id}`), and `stream-key`. **No GitHub App installation token, no fine-grained PAT, no OAuth-app token** is accepted on any of them. Required scope on the classic PAT is `read:audit_log`.

**Architectural implications** (M, sourced from the constraint above):

1. A Hulumi Pulumi run authed as a GitHub App **cannot** manage audit-log streaming destinations. If `OrgFoundation` configures audit-log streaming, it needs a separate classic-PAT credential — i.e. a **second `github.Provider` instance** in the Pulumi program, or a deliberate split where streaming is configured out-of-Pulumi by an operator.
2. The drift-classifier's GHEC-tier "high-fidelity" adapter that reads from `/enterprises/{enterprise}/audit-log` cannot share auth with the rest of the IaC program. It needs its own classic-PAT-only credential path. This contradicts Hulumi's preferred "least-privilege GitHub-App-only" stance and **the design must own this exception explicitly** rather than hide it.
3. There is no path on Free/Pro/Team to a programmatic audit-log even with a classic PAT — the endpoint family is enterprise-gated.

## Architecture Options

### Option A — Mirror the AWS three-package pattern (1:1)

**Layout**: `packages/baseline/src/github/` with `secure-repository.ts(+args+outputs)`, `org-foundation.ts(+args+outputs)`, sub-components (`org-rulesets.ts`, `org-actions.ts`, `org-oidc-template.ts`, `enterprise-security-analysis.ts`, `audit-log-streaming.ts`); `mappings/cis-github.ts`, `mappings/nist-ssdf-v1.1.ts`. Reuse `tier.ts` enum unchanged. Policy pack at `packages/policies/src/github/packs/hulumi-hardening.ts` (separate module to satisfy the one-PolicyPack-per-process invariant). Drift adapter at `packages/drift/src/adapters/github-audit-log.ts`.
**Trade-offs**: Highest internal consistency with AWS side; lowest cognitive load for the wedge persona who has Hulumi-AWS adopted. **Fully owns the two escape hatches** (code-security-configurations attach + audit-log-streams) inside the components — they appear as sub-resources with explicit "uses REST escape hatch" tagging. Drift adapter only works at GHEC.
**Auth complication**: `OrgFoundation` would need to accept _two_ `github.Provider` instances when audit-log streaming is configured — one GitHub-App-authed for everything else, one classic-PAT-authed _only_ for the audit-log-streams escape hatch (because of the GHEC API's classic-PAT-only constraint, [REST docs](https://docs.github.com/en/enterprise-cloud@latest/rest/enterprise-admin/audit-log)). This is unusual but explicit.
**Best fit**: GHEC customer who already runs Hulumi-AWS and wants idiomatic parity, accepts the dual-credential reality of audit-log streaming.

### Option B — Tier-aware split with first-class degraded path

Same package layout as A, but **drift adapters are explicitly split** into `github-audit-log-ghec.ts` (full fidelity, classic-PAT-authed, 4-adapter quorum equivalent) and `github-webhook-fallback.ts` (Team/Pro tier; subscribes to `branch_protection_rule`, `repository_ruleset`, `secret_scanning_alert_location` webhooks; push-model so no PAT needed). The classifier verdict carries a first-class `tier-degraded: true` field for non-GHEC customers. `OrgFoundation` audit-log-streaming sub-component **refuses to materialize** at non-GHEC tier rather than silently no-op'ing.
**Trade-offs**: One additional adapter to maintain and a more complex tier-decision tree. **Owns honestly** the truth that the wedge persona's company is more likely on Team than GHEC. Webhook-based fallback sidesteps the classic-PAT-only constraint entirely for the read path. Streaming-destination management remains GHEC + classic-PAT-only and is gracefully absent at Team.
**Best fit**: The brief's named persona — "platform engineer with ~30 hand-configured repos." Most companies at that scale are not GHEC. **Recommend as the default unless the user confirms GHEC.**

### Option C — Thin baseline + heavy policy pack

SecureRepository minimal (rulesets + security-and-analysis + Actions allowlist); `HulumiGithubHardeningPack` does the enforcement: rejects raw `github.Repository` without `securityAndAnalysis`, rejects providers with classic-PAT detected via Pulumi config inspection (best-effort — `token` field doesn't distinguish, so this becomes an env-var smell test), rejects rulesets without required-status-checks, requires `ActionsOrganizationOidcSubjectClaimCustomizationTemplate` to declare a `repository_owner:org/repo` shape (UNC6426 enforced at preview-time). Threat-model skill scenarios bind directly to policy IDs (G1/G2/G3...) the same way AWS H1/H2/H3/H4 do.
**Trade-offs**: Strongest preview-time enforcement story; matches the H1/H2/H3/H4 idiom in `packages/policies/src/index.ts`. Increases pull on Hulumi-tagged exemptions for legitimate edge cases; the classic-PAT detection rule will be best-effort, not surgical.
**Best fit**: Threat-model-focused customers who want skill scenarios (d), (c), (f), (e) to map cleanly to enforced policy IDs.

### Recommended composition

**B + C**, not either alone. B owns the tier reality of the wedge persona; C owns the threat-model-skill→policy-rule binding that makes the `/hulumi-threat-model` GitHub skill more than a doc generator. Option A's dual-credential audit-log story can be folded into B as a GHEC-only sub-feature.
