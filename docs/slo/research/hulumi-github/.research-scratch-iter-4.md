### Q3 — Webhook-fallback drift adapter at Team tier (Δ RESOLVED, H/M)

The iter-3 worst-case ("`repository_ruleset.edited` is itself GHEC-gated, invalidating the Team-tier fallback") **does not hold**. Per https://docs.github.com/en/webhooks/webhook-events-and-payloads:

| Event                                         | Repo / Org / Both                  | Deliverable at Team / Pro?                                                             |
| --------------------------------------------- | ---------------------------------- | -------------------------------------------------------------------------------------- |
| `branch_protection_rule`                      | Both                               | Yes                                                                                    |
| `repository_ruleset` (created/edited/deleted) | Both                               | **Yes** — repo-scoped ruleset edits fire on any tier where rulesets exist              |
| `secret_scanning_alert` / `_alert_location`   | Both                               | Yes if secret scanning enabled (private repos: GA on Team since 2024-03; public: free) |
| `dependabot_alert`                            | Both                               | Yes (free all tiers since 2022)                                                        |
| `code_scanning_alert`                         | Both                               | Public: yes; private: GHAS-licensed only                                               |
| `member` (collaborator + org-member)          | Both                               | Yes                                                                                    |
| `organization` (member_added/removed/etc.)    | **Org-only** subscription required | Yes at org scope                                                                       |

Implications (M): Option B's `github-webhook-fallback.ts` is structurally sound at Team. `OrgFoundation` must provision **two** webhook surfaces (org-level + per-repo), not one. Classifier verdict taxonomy must distinguish `feature-not-licensed` from `no-drift` for `code_scanning_alert` on private repos without GHAS.

### Q5/Q6 — Org-level security defaults at non-Enterprise tier (Δ RESOLVED, H)

Iter-3 raised whether Team/Pro orgs have any Pulumi-native path to org-level security defaults. **They do.** `github.OrganizationSettings` (https://www.pulumi.com/registry/packages/github/api-docs/organizationsettings/) exposes the six defaults: `advancedSecurityEnabledForNewRepositories`, `dependabotAlertsEnabledForNewRepositories`, `dependabotSecurityUpdatesEnabledForNewRepositories`, `dependencyGraphEnabledForNewRepositories`, `secretScanningEnabledForNewRepositories`, `secretScanningPushProtectionEnabledForNewRepositories` — alongside `defaultRepositoryPermission`, `membersCanForkPrivateRepositories`, `webCommitSignoffRequired`. Terraform `github_organization_settings` has identical snake_case parity.

**Three controls remain Enterprise-only** (handled by `EnterpriseSecurityAnalysisSettings`): `secretScanningValidityChecksEnabled`, `secretScanningPushProtectionCustomLink`, and the newer Code Security Configurations API. The escape-hatch list shrinks: only **two** REST escape hatches are needed in `OrgFoundation` — code-security-configurations attach + audit-log streams.

**Risk flag** (Δ NEW): GitHub's REST docs mark the flat `PATCH /orgs/{org}` `*_enabled_for_new_repositories` fields with an "endpoint closing down notice. Please use code security configurations instead." If those flat fields are removed before ship, `OrganizationSettings`-based components break at all tiers and the configurations-attach escape hatch becomes mandatory. Open question.

### Q8 — OIDC sub-claim safe defaults (Δ RESOLVED, H)

AWS, Azure, GCP, and GitHub all converge on three scoping axes (in increasing strictness): repo identity → ref/environment → `job_workflow_ref`. Per:

- AWS https://docs.aws.amazon.com/IAM/latest/UserGuide/id_jwt_creating_oidc_provider_github.html — `StringEquals` (not `StringLike`) on `sub`; bind to environment for prod.
- Azure https://learn.microsoft.com/en-us/entra/workload-id/workload-identity-federation-create-trust — environment-scoped subject per federated credential, no wildcards.
- GCP https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines — `attribute.repository` + `attribute.ref`/`attribute.environment`; for highest assurance use `attribute.job_workflow_ref`.
- GitHub https://docs.github.com/en/actions/deployment/security-hardening-your-deployments/about-security-hardening-with-openid-connect — recommends `job_workflow_ref` + `environment` or `ref` in the org template.

**Recommended Hulumi default** for `ActionsOrganizationOidcSubjectClaimCustomizationTemplate`:

```
repo:{org}/{repo}:job_workflow_ref:{org}/{repo}/.github/workflows/{workflow}@{ref}:environment:{environment}
```

with documented fallback `repo:{org}/{repo}:job_workflow_ref:...@{ref}:ref:refs/heads/main` for non-environment workflows.

UNC6426 abused trust policies whose `sub` used `StringLike repo:org/*:*` (~500 role ARNs across 275 AWS accounts). The default `repo:{org}/{repo}:*` shape Pulumi accepts is the exact UNC6426-vulnerable shape and **must not** be Hulumi's default. `HulumiGithubHardeningPack` should ship policy rule **G-OIDC-1**: reject any IAM/Entra/GCP-WIF trust whose `sub` condition uses `StringLike` or contains `*`.

## Architecture Options

The iter-3 recommendation **B + C** holds and is strengthened.

### Option B — Tier-aware split with first-class degraded path (RECOMMENDED DEFAULT)

Now confirmed feasible by Q3. `github-webhook-fallback.ts` subscribes to all six event types at Team with two named carve-outs:

1. Org-level events require a second webhook surface (`OrgFoundation` provisions both org-level and per-repo subscriptions).
2. Private-repo `code_scanning_alert` requires GHAS — classifier verdict carries `feature-not-licensed` as a first-class outcome distinct from `no-drift`.

GHEC-tier `github-audit-log-ghec.ts` adapter remains classic-PAT-only. Programs configuring audit-log streaming need a **second `github.Provider`** authed with a classic PAT (per https://docs.github.com/en/enterprise-cloud@latest/rest/enterprise-admin/audit-log). Owned explicitly, not hidden.

`OrganizationSettings` covers org-level security defaults at all tiers — escape-hatch surface is only two paths (code-security-configurations attach + audit-log streams), both REST.

**Best fit**: the wedge persona (~30 repos, hand-configured, Hulumi-AWS adopted, probably non-GHEC).

### Option C — Heavy policy pack (RECOMMENDED COMPANION TO B)

Gains a concrete first rule from Q8: **G-OIDC-1** = reject any IAM/Entra/GCP-WIF trust whose `sub` condition uses `StringLike` or contains `*`. Threat-model skill scenario (d) binds directly to G-OIDC-1.

Other rules from iter-3 unchanged: G-1 reject raw `github.Repository` without `securityAndAnalysis`; G-2 reject providers with classic-PAT smell test; G-3 reject rulesets without required-status-checks; G-4 require `ActionsOrganizationOidcSubjectClaimCustomizationTemplate` to declare a non-default sub shape (UNC6426 enforced at preview-time).

**Best fit**: customers who want preview-time enforcement and threat-model skill scenarios that map cleanly to enforced policy IDs (the H1/H2/H3/H4 idiom in `packages/policies/src/index.ts`).

### Option A — Mirror the AWS three-package pattern (1:1)

Folded into B as the GHEC-only execution path; not a standalone choice.
