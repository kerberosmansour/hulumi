# Synthesis — what the research means for the design

> Each paragraph closes with `the design must handle <X> because <source>` per `/slo-research`'s anti-vague-finding rule. Findings that cannot end that way are surfaced in `dossier.md` open-questions instead.

## Provider state

`pulumi-github` v6.13.0 (released 2026-04-24, a thin wrapper over `terraform-provider-github` v6.12.0) provides first-class IaC bindings for `RepositoryRuleset`, `OrganizationRuleset`, `OrganizationSettings`, `EnterpriseSecurityAnalysisSettings`, `ActionsOrganizationPermissions`, and `ActionsOrganizationOidcSubjectClaimCustomizationTemplate` — every primitive a `SecureRepository` + `OrgFoundation` mirroring the AWS pattern needs. Two structural gaps remain — Code Security Configurations attach/detach (`/orgs/{org}/code-security/configurations` and `/enterprises/{enterprise}/code-security/configurations`) and audit-log streams (`/enterprises/{enterprise}/audit-log/streams`). Terraform v6.12.0 has the same two gaps, so they are platform-wide REST escape hatches rather than Pulumi-specific deficits. **The design must encapsulate both REST escape hatches as named, tagged sub-resources of `OrgFoundation` (not silently no-op'd) because both Pulumi v6.13.0 and Terraform v6.12.0 release notes leave them unbound (Pulumi Registry `api-docs/`, terraform-provider-github releases page, retrieved 2026-04-25).**

## GHEC tier reality

GitHub's enterprise audit-log REST endpoints are GHEC-only and accept _only_ classic PATs carrying `read:audit_log`; GitHub App installation tokens, GitHub App user tokens, fine-grained PATs, and OAuth-app tokens are all rejected. Free/Pro/Team has no programmatic audit-log API at all; the Team-tier fallback is push-model webhooks delivering `branch_protection_rule`, `repository_ruleset`, `secret_scanning_alert`, `dependabot_alert`, `code_scanning_alert` (private repos: GHAS-licensed only), `member`, and org-only `organization`. **The design must expose `tier-degraded` and `feature-not-licensed` as first-class drift-classifier verdicts and accept two `github.Provider` instances on `OrgFoundation` for GHEC audit-log streams (one App-authed for everything else, one classic-PAT-authed exclusively for the streams escape hatch) because the auth-mode constraint and the tier gating are documented uniformly across all seven enterprise audit-log endpoints (GHEC `rest/enterprise-admin/audit-log`, retrieved 2026-04-25).**

## OIDC misconfiguration is the single highest-impact threat-model scenario

UNC6426 (March 2026) weaponized exactly the wildcard `repo:{org}/{repo}:*` subject shape that Pulumi accepts as the default `ActionsOrganizationOidcSubjectClaimCustomizationTemplate`, compromising approximately 500 vulnerable role ARNs across 275 AWS accounts (per CSA Labs and Unit 42 OH-MY-DC analyses). AWS, Azure, GCP, and GitHub guidance independently converge on a three-axis `repository → ref/environment → job_workflow_ref` scoping, with AWS specifying `StringEquals` (not `StringLike`) on `sub` and Azure federated credentials requiring environment-scoped subject without wildcards. **The design must ship a Hulumi default OIDC sub template using `job_workflow_ref` + `environment` (with `:ref:refs/heads/main` documented fallback) and a CrossGuard rule G-OIDC-1 that rejects wildcard or `StringLike` `sub` conditions on AWS/Azure/GCP trust policies, because the AWS IAM, Azure Entra Workload ID, GCP Workload Identity Federation, and GitHub Actions OIDC docs explicitly converge on the three-axis shape and UNC6426 demonstrated the cost of the wildcard default (AWS `id_jwt_creating_oidc_provider_github`, Azure `workload-identity-federation-create-trust`, GCP `workload-identity-federation-with-deployment-pipelines`, GitHub `about-security-hardening-with-openid-connect`, retrieved 2026-04-25).**

## License posture carries over unchanged

CIS GitHub Benchmark v1.2.0 (March 2026 cohort) ships under CC BY-NC-SA 4.0 plus CIS Terms of Use for Non-Member Products that forbid posting, sublicensing, and commercial redistribution. Apache-2.0 outbound stays compatible only if Hulumi cites by ID and upstream URL and never embeds control text — same posture as the existing CIS AWS Foundations Benchmark mapping at `packages/baseline/src/mappings/cis-aws.ts`. NIST SP 800-218 v1.1 (2022-02-03 final, public domain) is the safe SSDF mapping target — SP 800-218r1 IPD (v1.2) closed comments 2026-01-30 with no second draft published as of 2026-04-25. **The design must mirror `packages/baseline/src/mappings/cis-aws.ts`'s ID-only `as const` pattern for `cis-github.ts`, target SSDF v1.1 IDs, and annotate v1.2-draft awareness on individual practices that materially shift (without binding to v1.2 IDs until final), because CIS license terms forbid posting/sublicensing/commercial redistribution and SSDF v1.2 IDs may renumber on final publication (CIS Terms of Use for Non-Member CIS Products + CSRC News 2025-12-17, retrieved 2026-04-25).**

## Forward-looking deprecation risk on flat org-settings fields

GitHub has marked the `PATCH /orgs/{org}` flat `*_enabled_for_new_repositories` fields with an "endpoint closing down notice. Please use code security configurations instead" — exactly the fields `OrganizationSettings` writes today (`advancedSecurityEnabledForNewRepositories`, `dependabotAlertsEnabledForNewRepositories`, `secretScanningEnabledForNewRepositories`, etc.). If GitHub removes these fields before Hulumi-for-GitHub ships, every `OrgFoundation` deployment built directly on `OrganizationSettings` breaks, and the Code Security Configurations REST escape hatch flips from optional to mandatory at all tiers. The deprecation timeline is not yet sourced. **The design must encapsulate org-level security defaults behind an `OrgFoundation`-internal abstraction (not raw `OrganizationSettings` exposure) and pre-write the Code Security Configurations REST escape hatch as a switchable backend, because GitHub's deprecation notice on the flat fields is in-product as of 2026-04-25 and a removal date will appear without a migration window (Pulumi `OrganizationSettings` API docs + `pulumi-github` v6.13.0 schema, retrieved 2026-04-25).**

## Boundary discipline is supported by published frameworks

GitHub's Well-Architected NIST SSDF mapping is the cleanest infra-vs-workflow split available in 2026: `PO.2/PO.3/PO.4/PO.5/PS.1/PW.4/PW.5/PW.6/PW.7/RV.1` = Infrastructure-as-Code; `PS.2/PW.1/PW.2/PW.8/RV.2/RV.3` = Workflow. SLSA Source Track v1.2 reinforces the line: L1–L3 are repo/org configuration; L4 is the first level that mixes config with per-PR review behavior. OpenSSF Scorecard partitions identically: `Branch-Protection`, `Pinned-Dependencies`, `Token-Permissions`, `Webhooks` are configuration-driven; `Code-Review`, `Maintained`, `SAST`, `Vulnerabilities` are analysis-driven. The six contested boundary cases (a–f from the brief) all resolve cleanly against this mapping — (a)/(d)/(e) infra, (b)/(f) workflow (out of scope), (c) split with org-level enforcement infra and per-workflow source review workflow. **The design must adopt SSDF practice IDs (PO/PS/PW Infrastructure-as-Code subset) and Scorecard configuration-driven check IDs as the in-scope contract for the runbook, citing GitHub's own Well-Architected mapping as precedent, because no single framework draws the boundary as cleanly and citing one prevents drift back into appsec scope on contributor PRs (Well-Architected SSDF mapping + SLSA v1.2 source-requirements + scorecard.dev, retrieved 2026-04-25).**

## Threat-model skill scenarios

Demand-minus-supply ranking from 2025–2026 incidents — UNC6426 (March 2026), trivy-action March 2026 (75/76 tags compromised), tj-actions/changed-files (23k repos), Sysdig Shai-Hulud worm (November 2025), OpenAI Codex token-stealing branch-name injection (patched February 2026), Vercel April 2026 compromise — places (d) OIDC trust to cloud, (c) Actions supply-chain risk, (f) GitHub App / installation-token exposure, and (e) self-hosted runner risk as the highest-gap scenarios. By contrast, (a) org bootstrap baseline and (b) repository hardening are saturated by `safe-settings`, `cloudposse/terraform-github-repository`, and `mineiros-io/terraform-github-repository` README-level guidance. **The design must headline the four highest-gap scenarios — (d), (c), (f), (e) — in the `/hulumi-threat-model` GitHub skill rather than padding with (a) and (b), because saturated-supply scenarios produce skill outputs the user can already get from those modules' README files, while the four highest-gap scenarios anchor on named 2025–2026 incidents (Wiz prt-scan, Snyk Trivy, Sysdig Shai-Hulud, Unit 42 OH-MY-DC, CSA Labs OIDC trust chain abuse, retrieved 2026-04-25).**

## Architecture composition

The findings consolidate to a recommended composition of Option B (tier-aware split with first-class degraded path: GHEC adapter + webhook fallback adapter, drift-classifier verdict carries `tier-degraded` and `feature-not-licensed` as first-class outcomes) plus Option C (heavy `HulumiGithubHardeningPack` binding the four threat-model scenarios to enforced policy IDs G-OIDC-1 plus G-1..G-4 in the H1/H2/H3/H4 idiom). Option A's dual-credential audit-log story folds into B as a GHEC-only sub-feature. The single largest open question — the wedge persona's GitHub plan tier — is the only finding that cannot be closed from external sources and must be resolved with the user before `/slo-plan` runs. **The design must default to Option B + Option C composition, with Option A as a GHEC-only specialization rather than the default, because (i) the webhook-events catalog confirms delivery at all six event types relevant to drift on Team tier and (ii) the wedge persona's likely tier remains unconfirmed (Pulumi Registry resource set + GitHub `webhooks/webhook-events-and-payloads` + open question #1 in `dossier.md`, retrieved 2026-04-25).**
