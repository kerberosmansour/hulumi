# Milestone 5 — SLSA-L3 atomic three-package release of GitHub additions + launch readiness

Parent runbook: [docs/slo/completed/RUNBOOK-hulumi-github.md](../completed/RUNBOOK-hulumi-github.md). Read the runbook's Global Execution Rules + Global Entry Rules + lessons files for M1 / M2 / M3 / M4 before starting.

**Goal**: After M5, `@hulumi/baseline`, `@hulumi/policies`, `@hulumi/drift` are atomically re-released as v1.1.0 carrying the GitHub additions (M1–M4) with SLSA Build L3 attestation. Launch-readiness deliverables ship: three new examples (`secure-repository-smoke`, `org-foundation-smoke`, `github-drift-smoke`); four new cookbooks (one per threat-model scenario from M1: OIDC trust to cloud, Actions supply-chain, GitHub App tokens, self-hosted runners); component reference docs (M1–M4 components); README + AGENTS.md + `docs/why-hulumi.md` + `docs/getting-started.md` updates; weekly integration workflow extended to GitHub sandbox suite; verify-provenance docs updated; v1.1.0 CHANGELOG entry; and a v1.1 follow-up issue tracking the deferred GHEC-only surface (audit-log REST adapter, EnterpriseSecurityAnalysisSettings, audit-log streams, CIS GitHub Benchmark WorkBench-gated section numbers).

**Context**: Hulumi v1.0.0 (AWS) ships [`docs/slo/completed/RUNBOOK-hulumi.md`](../completed/RUNBOOK-hulumi.md) M5 already with SLSA-L3 attestation, npm trusted publishing, atomic three-package release, and the SCP / launch-readiness work. M5 of this runbook **extends** that pipeline to cover the GitHub additions; it does NOT re-design it. The atomic-three-package version bump is the existing convention from `docs/slo/completed/RUNBOOK-hulumi.md` § Runbook Metadata: `@hulumi/baseline`, `@hulumi/policies`, `@hulumi/drift` ship the same version on the same day. v1.1.0 is the right semver tier — features are added, no public API breaks. The cookbook and example surface mirrors the AWS-side at [`examples/secure-bucket-smoke/`](../../examples/secure-bucket-smoke/), [`account-foundation-smoke/`](../../examples/account-foundation-smoke/), [`drift-classify-smoke/`](../../examples/drift-classify-smoke/). The weekly integration workflow extension picks up the GitHub sandbox-org tests behind the existing `HULUMI_INTEGRATION=1` flag.

**Important design rule**: **No new code surface in M5.** This milestone is launch-readiness — examples, cookbooks, docs, release-pipeline extension, attestation verification. If a missing-feature gap is discovered during smoke testing, it is captured as a v1.1+ follow-up, NOT inserted into M5. The runbook's hard cap is five milestones — feature work past M4 is out of scope. The single exception is the AWS-vs-GitHub-side `lint:license-boundary` extension if M3 didn't capture every CIS GitHub Benchmark scaffold; that's a one-line edit, not a feature. Atomic three-package release means **all three** packages must build, attest, sign, and publish — partial success rolls back the entire release per the existing `release.yml` workflow's atomic-rollback discipline.

**Refactor budget**: `Surgical addition only.` New examples + new cookbook files + new component reference docs. Targeted edits to `README.md`, `AGENTS.md`, `docs/why-hulumi.md`, `docs/getting-started.md`, `CHANGELOG.md`, `.github/workflows/weekly-integration.yml`, `docs/cookbooks/README.md`, `docs/components/README.md`, `docs/threat-model-examples/README.md`. The release pipeline (`release.yml`) gets a one-line addition to the matrix if not already covering the three-package set; otherwise unchanged.
