# Milestone 2 — `IstioFoundation`

Parent runbook: [docs/slo/completed/RUNBOOK-hulumi-k8s.md](../completed/RUNBOOK-hulumi-k8s.md). Read the runbook's Global Execution Rules (especially **Rule 0 — the Hulumi-K8s scope contract** and **Rule 8 — version pinning**) + Global Entry Rules + M1's lessons before starting.

**Goal**: After M2, `@hulumi/k8s-baseline.IstioFoundation` ships as a `pulumi.ComponentResource` that bundles three version-pinned Helm releases (istiod with `pilot.cni.enabled=true`, istio-cni DaemonSet with Fargate-exclusion affinity, istio-ingressgateway as ClusterIP) into one declaration with PSA-baseline-clean defaults — closes [#39](https://github.com/kerberosmansour/hulumi/issues/39) and fully closes [#42](https://github.com/kerberosmansour/hulumi/issues/42) (the wrapper's `daemonSet: true` arg now serves a real consumer in production code paths). The component exposes a `MeshFoundation` ref shape (`ingressGatewayServiceAccountName`, `ingressGatewayReleaseName`, `istiodReleaseName`, `cniReleaseName`) that M3's `AlbMeshedHttpEntrypoint` consumes.

**Context**: The cookbook at [`docs/cookbooks/psa-baseline-istio-sidecar.md`](../cookbooks/psa-baseline-istio-sidecar.md) (shipped in #45) walks through the underlying interaction by hand. M2 collapses that ~150-line cookbook recipe to a single component invocation with the same semantics: `pilot.cni.enabled=true` on istiod, istio-cni DaemonSet via `HardenedHelmRelease(daemonSet: true)`, ingress gateway in a separate namespace `istio-ingress` (per design record open question Q4 → bias accepted: separate namespace). Version is a single arg applied to all three releases — the version-skew gotcha that motivates packaging is collapsed to "one wrong number, all three releases catch it."

**Important design rule**: **`IstioFoundation` orders its three Helm releases via explicit `dependsOn`**: `istio-cni` → `istiod` → `ingressGateway`. The cookbook explains why: if `istiod` lands first, the sidecar-injector ConfigMap gets written and namespaces start admitting pods before the CNI is Ready, and the racing first wave still hits `istio-init` until the CNI catches up. The component encodes this order; consumers don't have to know it. The order is the abstraction's reason to exist.

**Refactor budget**: `Surgical addition only.` New `packages/k8s-baseline/src/istio-foundation*` files + tests. Modifies `packages/k8s-baseline/src/index.ts` (re-export), `packages/k8s-baseline/src/compatibility.ts` (add `istiod`, `istio-cni`, `istio-ingressgateway` chart entries to `TESTED_VERSIONS` table), `packages/k8s-baseline/COMPATIBILITY.md` (document the version table), `packages/k8s-baseline/src/hardened-helm-release.ts` (extend the `chartClass` enum with `"istio"` and add the 480_000ms default timeout), `docs/slo/completed/RUNBOOK-hulumi-k8s.md` Milestone Tracker.

## Out of Scope / Must Not Do

- No `AlbMeshedHttpEntrypoint` — M3.
- No mesh alternatives. Linkerd / Cilium service mesh / Consul Connect / App Mesh stay out of scope (Rule 0).
- No automatic Fargate-profile detection. The `excludeFargate` flag is explicit per the design record's M1-committed decision.
- No `EnvoyFilter` / `WasmPlugin` / `Telemetry` / Istio addons (Kiali, Jaeger, Grafana). The component installs the three load-bearing pieces; addons are workload/observability decisions consumers make per-cluster.
- No multi-cluster mesh / Istio multi-primary topology. Single-cluster only at v1; revisit on demand.
- No CRD installation outside what istiod's chart manages. If a user pinned to an Istio version where CRDs are split into a separate chart, document the gap in `COMPATIBILITY.md` rather than installing a fourth chart silently.
