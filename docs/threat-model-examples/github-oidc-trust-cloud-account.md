# Threat Model — GitHub Actions OIDC trust to a cloud account — three-axis sub claim, UNC6426 mitigation

> Generated by the [`/hulumi-threat-model`](https://github.com/kerberosmansour/hulumi) Claude Code skill (exemplar — handcrafted target-shape reference). This document cites CIS GitHub Benchmark v1.2.0, NIST SSDF v1.1, OpenSSF Scorecard, MITRE ATT&CK T1195, GitHub Well-Architected SSDF, and CSA CCM v4.1 by **identifier only**. See [`docs/mappings/licensing.md`](https://github.com/kerberosmansour/hulumi/blob/main/docs/mappings/licensing.md) for the IDs-only policy.

## Scenario

A platform engineer wires GitHub Actions to assume a role in AWS / Azure / GCP via OIDC federation. The cloud-side trust policy must validate the GitHub-issued token's `sub` claim against a three-axis shape (`repo:{org}/{repo}:job_workflow_ref:{org}/{repo}/.github/workflows/{workflow}@{ref}:environment:{environment}`). UNC6426 (March 2026, ~500 vulnerable role ARNs across 275 AWS accounts per CSA Labs / Unit 42 OH-MY-DC analyses) weaponized exactly the wildcard `repo:{org}/{repo}:*` shape that Pulumi's GitHub provider accepts as default. Hulumi's `OrgFoundation` (M2) ships the safe template; the M3 `G_OIDC_1` policy declaratively rejects `StringLike` / wildcard `sub` conditions on AWS / Azure / GCP trust policies.

## Actors

- Platform Engineer (trusted)
- AI Coding Agent (trusted, authoring IaC via Claude Code)
- External Attacker controlling a fork or a separate repo in the same org
- Compromised third-party Action (semi-trusted, via supply-chain compromise)

## Assets

- GitHub Actions OIDC issuer URL (`token.actions.githubusercontent.com`)
- Cloud-side IAM role assumed by the workflow
- `sub` claim shape in the cloud trust policy
- `ActionsOrganizationOidcSubjectClaimCustomizationTemplate` (org-level IaC)
- Production cloud resources downstream of the assumed role

## Threats (STRIDE)

| Type | Name | Description | Controls |
|---|---|---|---|
| S | OIDC sub-claim wildcard impersonation (UNC6426) | An attacker who can run any workflow in the same org requests an OIDC token; the cloud trust policy uses `StringLike` on `sub` with wildcard `repo:{org}/{repo}:*`; the attacker's token is accepted and the role is assumed. | CCM:IAM-10, NIST-SSDF-v1.1:PO.5, OpenSSF-Scorecard:Token-Permissions, MITRE-ATTCK:T1195, GitHub-Well-Architected:PO.3 |
| T | Trust policy tampering via console | An admin clicks in the AWS / Azure / GCP console and broadens the OIDC trust policy's `sub` condition for a one-off use case. | CCM:IAM-06, NIST-SSDF-v1.1:PW.7, GitHub-Well-Architected:RV.1, CIS-GitHub-v1.2.0 |
| I | Information disclosure via over-scoped role | The role assumed by the GitHub Actions workflow has broader permissions than the workflow needs. | CCM:IAM-07, NIST-SSDF-v1.1:PO.3, OpenSSF-Scorecard:Token-Permissions, GitHub-Well-Architected:PW.5 |
| E | Privilege escalation via job_workflow_ref spoofing | Without an `environment` axis, the trust policy cannot distinguish prod-deploy from non-prod-deploy callers. | CCM:IAM-10, NIST-SSDF-v1.1:PO.5, GitHub-Well-Architected:PO.3, MITRE-ATTCK:T1195 |
