# Threat Model — GitHub Actions supply-chain risk — third-party Action ingestion, SHA-pinning, pwn-request, cache poisoning

> Generated by the [`/hulumi-threat-model`](https://github.com/kerberosmansour/hulumi) Claude Code skill. This document cites CSA CCM v4.1, CIS AWS Foundations Benchmark v5.0.0, NIST SP 800-53 Rev 5, NIST SP 800-218 / 218A, and MITRE ATLAS v5.1 by **identifier only**. See [`docs/mappings/licensing.md`](https://github.com/kerberosmansour/hulumi/blob/main/docs/mappings/licensing.md) for the IDs-only policy.

## Scenario

A platform engineer adopts third-party Actions in CI workflows. Recent incidents - trivy-action March 2026 (75/76 tags compromised), tj-actions/changed-files (~23k repos affected), Sysdig Shai-Hulud worm (November 2025), Wiz prt-scan campaign, and AI-scaled fork-PR pwn-request campaigns - have all exploited adjacent primitives: workflows referencing Actions by mutable tag, orgs without an allow-list policy, privileged workflows checking out untrusted PR code, and release/deploy jobs that expose secrets or OIDC tokens to code from the wrong trust boundary. Hulumi's `OrgFoundation` (M2) ships the SHA-pin-required Actions allowlist policy at startup-hardened tier; M3's `HulumiGithubHardeningPack` enforces this declaratively at preview-time; the workflow-governance linter gives consumers a local CI gate for workflow YAML drift.

## Actors

- Platform Engineer (trusted, configures the org-level allowlist)
- Workflow Author (semi-trusted, picks Actions from the allow-listed set)
- Compromised Maintainer of a third-party Action (untrusted, primary attacker)
- External PR Attacker (untrusted, opens AI-generated fork PRs at scale)
- AI Coding Agent (trusted, authoring workflows; could be tricked into citing mutable tags)

## Assets

- Workflow YAML files referencing third-party Actions
- Organization-level Actions allowlist (`ActionsOrganizationPermissions`)
- Repository secrets accessible to the workflow
- OIDC token minting permission (`id-token: write`)
- Build artifacts produced by the workflow
- Cache entries in `actions/cache`

## Recommended Hulumi Components

- `@hulumi/baseline.github.OrgFoundation` — Shipped in Hulumi v1.1.0 M2. Provisions the org-level `ActionsOrganizationPermissions` with `shaPinningRequired: true` at startup-hardened tier; configures the Actions allowlist (selectedActionsPatterns).
- `@hulumi/policies.github.HulumiGithubHardeningPack` — Shipped in Hulumi v1.1.0 M3. H1 declaratively rejects raw `github.Repository` not wrapped in `SecureRepository`; H2/H3/H4 enforce ruleset, OIDC, and tier-monotonicity invariants.
- `@hulumi/drift.adapters.GithubWebhookFallbackAdapter` — Shipped in Hulumi v1.1.0 M4. Detects org-level allowlist drift via the `repository_ruleset` and `organization` webhook events; surfaces `tier-degraded` honestly for non-GHEC tiers.
- `scripts/workflow-governance-lint.mjs` — Shipped in Hulumi v1.4.0; WF_PR_1 added after the 2026-05-25 pwn-request review. Fails privileged `pull_request_target` / `workflow_run` workflows that check out attacker-controlled head code, alongside SHA pinning, token-permission, CODEOWNERS, and protected-environment checks.

## Hulumi Repository Recommendations

- Keep the current no-`pull_request_target` posture for code execution. Metadata-only workflows may use `pull_request_target`, but `WF_PR_1_NO_UNTRUSTED_HEAD_CHECKOUT` now fails any privileged PR-head checkout.
- Keep PR CI on `pull_request` with top-level `permissions: contents: read`; do not add secrets, write-capable `GITHUB_TOKEN`, or cloud deploy credentials to PR-triggered jobs.
- Keep OIDC-bearing jobs out of fork-PR execution. The CI pre-tag dry run is now split into `release-artifacts-dry-run` for PRs with read-only permissions and `attestation-dry-run` for `main` pushes with `id-token: write`.
- Verify repository and organization settings that are not visible in workflow YAML: require full-length SHA pins, selected Actions allowlist, all-external-contributor workflow approval, read-only default workflow permissions, disabled Actions-created PR approvals, protected release environments, secret scanning, and push protection.
- Keep `.github/workflows/` in CODEOWNERS and require code-owner review on workflow, package-manager, release, and script changes.
